# 🧠 Ключевые концепции проекта

В этом руководстве собраны пояснения к основным механизмам и логике, используемым в данном проекте Ansible.

---

### 1. `infra_state` (Управление жизненным циклом)
Это пользовательская переменная, которая позволяет одной командой развернуть или полностью удалить конфигурацию.
- **`present`**: (По умолчанию) Установка пакетов, создание пользователей, запуск служб.
- **`absent`**: Удаление пакетов, удаление пользователей, остановка служб.
*Пример использования:* `ansible-playbook site.yaml -e "infra_state=absent"`

### 2. Ключевые переменные конфигурации
Эти переменные определяют параметры установки и настройки в `defaults/main.yml` каждой роли:
- **`linux_packages` / `windows_packages`**: Список ПО для установки.
- **`linux_user_name` / `windows_user_name`**: Имя создаваемой учетной записи.
- **`linux_config_src` / `windows_config_src`**: Исходный файл конфигурации.
- **`linux_config_dest` / `windows_config_dest`**: Путь назначения на целевом хосте.

### 3. Условная логика: `ternary` и `when`
В проекте используются два способа реализации логики в зависимости от `infra_state`.

- **Фильтр `ternary` (Внутри одного модуля):**
  Компактный аналог `if-else`. Используется, когда один модуль поддерживает оба состояния.
  *Синтаксис:* `{{ (условие) | ternary('ДА', 'НЕТ') }}`
  *Пример:* Переключение состояния службы (`started`/`stopped`) внутри модуля `service`.
- **Оператор `when` (Переключение между модулями):**
  Используется, когда для реализации `absent` требуется другой модуль.
  *Пример:* Модуль `copy` не умеет удалять файлы, поэтому для `present` используется `copy`, а для `absent` — модуль `file` с параметром `state: absent`.

### 4. `import_tasks` vs `include_tasks` (Статика vs Динамика)
Определяет, как и когда Ansible подключает файлы задач.
- **`import_tasks` (Статический):** Файл "вклеивается" в плейбук в момент его ЗАГРУЗКИ.
    - *Плюсы:* Работают теги, Ansible заранее знает обо всех задачах.
    - *Минусы:* Нельзя использовать переменные в именах файлов или циклы (`loop`).
- **`include_tasks` (Динамический):** Файл подключается во время ВЫПОЛНЕНИЯ (как вызов функции).
    - *Плюсы:* Можно использовать переменные в именах и циклы.
    - *Минусы:* Задачи внутри файла не видны до тех пор, пока до них не дойдет очередь.

### 5. `notify` и `handlers` (Умное реагирование)
Механизм для перезапуска служб только при необходимости.
- **`notify`**: "Сигнальная кнопка" внутри задачи. Нажимается ТОЛЬКО если задача что-то изменила (статус `changed`).
- **`handlers`**: Список обработчиков, которые срабатывают ОДИН раз в самом конце плейбука, если на них поступил сигнал через `notify`.
*Преимущество:* Если вы правите 5 конфигов одного сервиса, он перезагрузится всего один раз в конце.

### 6. `direction` (Направление трафика в Firewall)
Используется для настройки правил доступа (модули `ufw`, `win_firewall_rule`).
- **`incoming` (in):** Трафик, идущий ИЗВНЕ на ваш сервер.
- **`outgoing` (out):** Трафик, идущий С ВАШЕГО сервера во внешнюю сеть.

### 7. `insertbefore` / `insertafter` (Позиционирование строк)
Параметры модуля `lineinfile` для точной вставки текста в файл.
- **`insertbefore: BOF`**: Вставить строку в самое начало файла (Beginning of File).
- **`insertafter: EOF`**: Вставить строку в самый конец файла (End of File).
- **`insertbefore: '^Pattern'`**: Найти строку по регулярному выражению и вставить новую ПЕРЕД ней.

### 8. `templates` (Динамические конфигурации)
Механизм создания файлов на основе шаблонов Jinja2.
- **Директория**: `templates/` (файлы с расширением `.j2`).
- **Суть**: В отличие от простого копирования (`copy`), шаблоны позволяют подставлять значения переменных и использовать логику прямо внутри файла.
- **Модули**: `ansible.builtin.template` (Linux) и `ansible.windows.win_template` (Windows).
*Пример:* Генерация конфигурации сервера с уникальным именем хоста.
